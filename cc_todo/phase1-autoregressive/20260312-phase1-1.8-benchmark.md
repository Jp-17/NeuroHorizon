- **数据根目录**：`data/processed/`
- **bin size**：20ms

**关键 Bug 修复**：

1. **Spike binning 时间戳 bug**（严重）：
   - 问题：`bin_spike_events(ts, ui, n_units, start, end, ...)` 中 start 使用了绝对时间（如 97.627s），但 `torch_brain.Dataset.get()` 返回的 spike timestamps 是相对时间（0 到 window_length）
   - 症状：预测窗口 spike counts 全为 0（所有 spikes 被 clip 到 bin 0）
   - 修复：使用 `data.start`（=0）和 `data.end`（=window_length）作为 binning 参考

2. **IBL-MtM / Neuroformer 数据泄漏 bug**（严重）：
   - 问题：causal mask 允许当前位置 attend 到自身，而预测窗口位置的输入包含了 target spike counts
   - 症状：IBL-MtM 2 epochs 即达 fp-bps=2.15（正常值应 < 0.2），R²=0.90
   - 修复：在 forward() 和 compute_loss() 中将预测窗口输入置零（与 NDT2 的 MAE 方式一致）
   - 修复后：IBL-MtM 2 epochs fp-bps=0.11（合理）

**Sanity Check 验证结果**（2 epochs, pred_window=250ms）：

| IBL-MtM | 10.7M | 0.3252 | 0.3179 | 0.112 | 0.221 |

**20 epochs 验证结果**（NDT2, pred_window=250ms）：

#### 4.9 进一步执行：variable-length tokenization / 合法 padding 修复后，旧 near-zero 结果失效

在 4.8 的基础上，我继续按 Claude 的补充意见往数据路径里深挖，结果确认前一轮 faithful NDT2 还有两个更隐蔽、但影响更大的 fidelity 问题：

1. **旧 bridge 把所有 session 都强行扩成全局 `channel_capacity=72` 的 flat token 序列。**
   - 这和上游 `serve_tokenized_flat` 不一致。
   - Perich-Miller 10 sessions 的真实 token 数应随 session unit 数变化：
     - `71 units -> 9 tokens/step`
     - `52 units -> 7 tokens/step`
     - `41 units -> 6 tokens/step`
     - `38/37 units -> 5 tokens/step`
     - `19/18 units -> 3 tokens/step`

2. **旧 `pad_token=20` 会在 mixed-session batch 上把 padded `position` 写成越界索引。**
   - 当前 `max_spatial_tokens=9`
   - 一旦修正 variable-length tokenization，full-data mixed-session 训练就会触发 CUDA `indexSelectLargeIndex` / `device-side assert`

### 本轮代码修正

1. `tokenize_spike_counts()` 改为按每个 session 的真实 `n_units` 生成 flat tokens，不再硬扩到全局 72 channels
2. `collate_faithful_ndt2_batch()` 改为使用合法 pad token 做 batch padding
3. `create_faithful_ndt2_model()` 增加 `pad_token < max_spatial_tokens` 保护
4. 默认配置进一步对齐上游 `flat_enc_dec/f8`：
   - `spike_embed_style=token`
   - `dropout=0.1`
   - `lr_ramp_steps=100`
   - `lr_decay_steps=2500`
5. 同时保留 CLI 参数，允许显式切回旧 `project` embedding / schedule 做对照

#### 4.11 为什么实现打通后结果仍然显著不成立（当前假设）

下面这些不是已经证实的结论，而是当前最可能的解释，后续需要按优先级逐条验证：

1. **训练目标和 benchmark 目标并不一致。**
   - 当前 faithful NDT2 保留的是上游 `ShuffleInfill` 训练路径
   - 但最终 benchmark 衡量的是固定 `obs_window -> pred_window` 的 forward prediction
   - 这更像是 objective mismatch，而不是纯实现 bug

2. **训练采样分布仍与 NeuroHorizon 主线不同。**
   - 当前 faithful NDT2 虽然已经统一了数据源 / split / eval / metric
   - 但 train-time 仍不是 NeuroHorizon 的 `RandomFixedWindowSampler`
   - 这不再被视为“必须完全消除的问题”，但仍可能带来性能差异

3. **NDT2 的原始 inductive bias 可能并不适合当前 canonical forecasting benchmark。**
   - faithful reproduction 解决的是“是不是原模型”
   - 不保证解决“是不是当前 benchmark 的合适模型”

4. **mixed-session flat token batching 可能仍然是结构性困难。**
   - Perich-Miller 10 sessions 的 token 数从 `3` 到 `9` 不等
   - variable-length tokenization 和 padding 虽已修复
   - 但 mixed-session 训练本身仍可能让 NDT2 更难学到稳定表示

5. **当前 bridge 仍不是上游 repo 的原生训练生态。**
   - 现在已经是“上游核心模型 + compatibility bridge”
   - 但并不是直接在上游原生 dataset/task pipeline 中训练
   - 因此仍可能存在非致命但会影响性能的 subtle mismatch

6. **上游 optimizer/scheduler 直迁失败，说明问题不太像单纯调参不足。**
   - `configure_optimizers() + accumulate=16 + warmup-cosine` 已经试过
   - 结果没有变好，反而明显更差
   - 所以继续围绕 warmup / accumulate 做局部调参，预期收益有限

### 接下来还需要做什么

1. 先解释 `ShuffleInfill` 与当前 canonical forward prediction benchmark 之间的 objective mismatch
2. 再判断是否需要 NDT2 专用 adaptation，才能在不破坏统一评估协议的前提下成立
3. 只有在 250ms 的失败原因更清楚之后，才考虑扩展到 500ms / 1000ms
4. IBL-MtM / Neuroformer faithful 正式训练继续后置，不在当前 protocol 仍不稳定时展开
