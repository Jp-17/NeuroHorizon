# 2026-03-16 1.8.3 Benchmark 审计报告（Codex）

## 审查结论

当前 `plan.md` 中 **1.8.3「模型适配与对比实验」不能被视为有效的 benchmark 对比实验**。从代码实现、数据使用、评估口径和结果传播四个维度看，这一轮实验更准确的定位应是：

> **3 个“受 NDT2 / Neuroformer / IBL-MtM 启发的自定义 Transformer baseline”内部探索实验**

而不是“对原始 benchmark 模型完成适配并进行同条件公平对比”。

因此，当前所有基于 1.8.3 得出的强结论，例如“NeuroHorizon 在所有预测窗口上优于 NDT2 / Neuroformer / IBL-MtM”“Neuroformer 是最强竞争者”“并行预测方式不如自回归”等，都应降级为**暂不成立 / 需要重做实验后再判断**。

下面是按严重度排序的审查发现。

---

## Findings

### 1. [致命] 这不是对 NDT2 / Neuroformer / IBL-MtM 的真实适配，而是 3 个项目内重写的简化模型

**证据**：

- `plan.md` 将 1.8.3 描述为“各模型适配代码（neural_benchmark/adapters/）”和“模型适配与对比实验”，并明确写成 benchmark 对比，`cc_core_files/plan.md:531-585`。
- 任务记录也把这些实现写成 `NDT2Wrapper / IBLMtMWrapper / NeuroformerWrapper`，`cc_todo/phase1-autoregressive/20260312-phase1-1.8-benchmark.md:136-152`。
- 但真实代码里：
  - `neural_benchmark/adapters/ndt2_adapter.py:22-27` 明确写着 `Simplified NDT2 model wrapper`，并说明 “Instead of using the full BrainBertInterface, we build a minimal NDT2-style model”。
  - `neural_benchmark/adapters/ibl_mtm_adapter.py:17-29` 实际是一个项目内手写的 `IBLMtMWrapper`，没有调用 IBL-MtM 原始训练/推理栈。
  - `neural_benchmark/adapters/neuroformer_adapter.py:14-24` 也是 `Simplified Neuroformer-style model`，并未使用 Neuroformer 原始 tokenization / discrete spike generation 方案。
- `benchmark_train.py` 训练时只调用这 3 个项目内 wrapper，`neural_benchmark/benchmark_train.py:23-29,33-37,135-138`。也就是说，clone 下来的 benchmark 仓库并没有真正成为实验主体。

**问题本质**：

这不是“适配已有模型到当前数据接口”，而是“参考论文/仓库思想，重新实现 3 个简化版 baseline”。这两者的学术含义和说服力完全不同。

**进一步的问题**：

这些 wrapper 只吃 `spike_counts`，没有把 session identity / unit identity 作为模型输入真正建模：

- `BenchmarkDataset` 虽然返回了 `session_id` 和 `unit_ids`，`neural_benchmark/adapters/base_adapter.py:203-216`；
- 但 3 个模型 forward 都只接收 `spike_counts`，见：
  - `neural_benchmark/adapters/ndt2_adapter.py:74-110`
  - `neural_benchmark/adapters/ibl_mtm_adapter.py:108-119`
  - `neural_benchmark/adapters/neuroformer_adapter.py:99-110`

这意味着跨 session 训练时，不同 session 的第 0 列、第 1 列……在模型看来只是“同一批固定通道槽位”，而不是不同 session 的不同神经元身份。这对跨 session 10-session 对比尤其不合适，也进一步说明这不是对原模型公平、忠实的适配。

**结论**：

1.8.3 目前最多只能叫：

> “3 个受 benchmark 启发的简化版内部 baseline”

不能叫：

> “NDT2 / Neuroformer / IBL-MtM benchmark 对比结果”

---

### 2. [致命] 文档宣称在相同 test intervals 上统一评估 4 个指标，但代码实际上只做了 validation 上的部分指标评估

**文档宣称**：

- `plan.md` 明确写着“所有模型在完全相同的 test intervals 上评估”，并统一计算 `fp_bps()`, `r2_score()`, `psth_r2()`, `PoissonNLLLoss`，`cc_core_files/plan.md:562-565`。
- 1.8.3 任务说明也把评估指标写成 “fp-bps / R² / PSTH-R² / Poisson NLL”，`cc_core_files/plan.md:581-585`。

**代码实际做的事**：

- `benchmark_train.py` 虽然创建了 `test_dataset`，但整个训练流程只构造 `val_loader`，没有 `test_loader`，`neural_benchmark/benchmark_train.py:40-63,119-127`。
- 所有周期性评估都发生在 `val_loader` 上，`neural_benchmark/benchmark_train.py:184-218`。
- 最终 `results.json` 只保存：`best_val_fp_bps`、`history` 等 validation 产物，`neural_benchmark/benchmark_train.py:225-237`。
- benchmark 统一评估函数 `evaluate_model()` 实际只返回 `fp_bps / r2 / per_bin_fp_bps / poisson_nll`，并没有 `psth_r2`，`neural_benchmark/adapters/base_adapter.py:247-302`。
- 结果文件键也印证了这一点：`results/logs/phase1_benchmark_*_250ms/results.json` 只有 `model / pred_window / obs_window / n_params / best_val_fp_bps / total_time_s / history`，根本没有 PSTH-R² 或 test 指标。

**直接后果**：

- “完全相同的 test intervals 上评估”这个说法是**错的**。
- “benchmark 也完成了 PSTH-R² 评估”这个说法是**错的**。
- 当前 1.8.3 的最终数字，本质上是**基于 validation 的 model selection 结果**，不是 held-out test result。

这会直接导致结论偏乐观，尤其是当这些 best validation 数字又被拿去和别的实验的独立评估值做横向对比时，偏差会被继续放大。

### 3. [高] benchmark 与 NeuroHorizon 1.3.4 并不在“完全一致条件”下比较，数据采样器和样本分布都不同

**文档宣称**：

- `results.md` 在 1.3.4 benchmark 对比处明确写着“条件完全一致（10 sessions、500ms obs_window、连续训练、同一数据划分）”，`cc_core_files/results.md:699-701`。

**实际代码差异**：

1. benchmark 使用的是**固定 50% overlap 滑窗**：
- `BenchmarkDataset` 在每个 interval 内用 `t += window * 0.5` 切窗，`neural_benchmark/adapters/base_adapter.py:127-137`。

2. NeuroHorizon continuous eval 使用的是 **RandomFixedWindowSampler**：
- `eval_phase1_v2.py` continuous eval 直接走 `RandomFixedWindowSampler`，`scripts/analysis/neurohorizon/eval_phase1_v2.py:123-127`。
- 这个 sampler 每个 interval 只取近似 `floor(length / window_length)` 个窗口，并带随机 offset，不是 50% overlap 全覆盖，`torch_brain/data/sampler.py:13-24,45-72,78-137`。

**数字证据**：

- 1.8.3 文档写 benchmark valid 样本数为：
  - 250ms: 1588
  - 500ms: 1127
  - 1000ms: 655
  见 `cc_todo/phase1-autoregressive/20260312-phase1-1.8-benchmark.md:118-122`
- NeuroHorizon `eval_v2_results.json` 中 continuous eval 的 `n_samples` 实际只有：
  - 250ms: 841
  - 500ms: 607
  - 1000ms: 380

这不是“同一批 valid/test 样本上比较”，而是**不同采样分布下的两个验证结果**。

**结论**：

即便 split 都来自 `get_sampling_intervals(split)`，也不能据此说 benchmark 和 NeuroHorizon 是“完全相同条件”。

当前更准确的说法应是：

> “二者使用了同一数据集与相同的 split 来源，但 continuous evaluation 的切窗策略不同，因此数值不构成严格可比的 apples-to-apples benchmark。”

### 4. [高] 结果汇总把 best checkpoint 的 fp-bps 与最后一个 epoch 的 R² / NLL 混在一起，导致“Best R² 汇总”不成立

**问题位置**：

### 5. [中高] null model 并没有真正统一，`统一 null model` 的说法不准确

**文档宣称**：

- `plan.md` 写的是共用同一个 null model，且来自 `compute_null_rates()`，`cc_core_files/plan.md:562-565`。
- `results.md` 也写成了“fp-bps（统一 null model）”，`cc_core_files/results.md:617`。

**代码实际**：

- NeuroHorizon 1.3.4 的 null rate 是通过 `compute_null_rates(train_dataset, model, model.bin_size)` 计算，`examples/neurohorizon/train.py:224-231`，`scripts/analysis/neurohorizon/eval_phase1_v2.py:357-360`。
- benchmark 则使用了自定义的 `compute_benchmark_null_rates()`，它是对 `BenchmarkDataset` 的重叠滑窗样本逐窗累计，`neural_benchmark/adapters/base_adapter.py:220-244`。

**问题**：

这两套 null 计算不但不是同一个实现，而且 benchmark 版本还是在**50% overlap 的窗口样本上**估计均值发放率，会改变时间点的加权方式。即便最终数值可能接近，也不能再称作“统一 null model”。

这是一个口径问题，不一定会像前几条那样直接推翻全部结果，但它会削弱 fp-bps 的可比性，尤其是在你已经同时改变了采样器和样本分布的前提下。

## 建议的修正顺序

### A. 立即修正结论层表述（必须）

把所有 “benchmark 对比结论” 降级为保守说法，例如：

- 将 “NDT2 / Neuroformer / IBL-MtM benchmark” 改成：
  - `受 NDT2 / Neuroformer / IBL-MtM 启发的简化版 baseline`
- 将 “NeuroHorizon 在所有窗口上优于 benchmark” 改成：
  - `在当前项目内自定义 baseline 上取得更高 validation fp-bps，但由于适配与评估口径未完全对齐，暂不能据此宣称优于原始 benchmark 模型`。

### B. 重做 benchmark pipeline（必须）

至少补齐下面 4 点后，结果才有资格被当作正式 benchmark：

1. **真实适配原模型**：
   - NDT2 应调用其真实模型/配置/训练接口；
   - Neuroformer 应尽可能保留其原始 tokenization / 自回归生成范式；
   - IBL-MtM 应保留其原始输入组织和训练目标；
   - 如果做不到，就明确承认这些只是 `custom baselines`，不要再借用 benchmark 名称。

2. **显式建模 unit / session identity**：
   - 至少不能让跨 session 数据共享同一个固定 channel 槽位语义；
   - 否则 10-session 对比会天然扭曲。

3. **统一评估协议**：
   - 用 validation 只做 checkpoint selection；
   - 最终报告只写 held-out test；
   - benchmark 与 NeuroHorizon 必须使用同一 split、同一 sampler、同一 null-model 计算逻辑、同一指标脚本。

4. **补齐 PSTH-R²**：
   - 如果文档要写 8 方向 trial-aligned PSTH-R²，就必须给 benchmark 也接上相同 trial-aligned eval；
   - 否则直接删掉这项宣称，不要在 plan / results / 图表里继续出现。

### C. 修正结果存储协议（强烈建议）

`results.json` 至少应同时保存：

- `best_epoch`
- `best_val_fp_bps`
- `best_val_r2`
- `best_val_nll`
- `best_metrics_at_checkpoint`
- `final_epoch_metrics`
- `test_metrics`

否则后续文档很容易再次出现“fp-bps 取 best，但 R² 取最后一次”的混写问题。

### D. 回收下游传播（必须）

至少应回收或改写这些位置的强结论：

- `cc_core_files/results.md:667-726`
- `scripts/analysis/neurohorizon/phase1_benchmark_compare.py`
- `scripts/analysis/neurohorizon/phase1_14_15_visualize.py`

如果短期内不重做实验，我建议直接在这些文档/脚本旁边增加醒目的 caveat：

> 这些结果来自项目内简化 baseline，不代表原始 benchmark 模型的正式复现结果。

## 最终判断

**当前 1.8.3 的主要问题不是“实现有些小瑕疵”，而是“任务命名、实验定位、评估口径和最终叙事都比实际证据走得更远”。**

如果你的目标是内部探索，那么这一轮工作是有价值的：它告诉你 “简化的 GPT-style / causal encoder / MAE-style baseline 在当前数据接口下大概能到什么量级”。

但如果你的目标是对外写成 benchmark 对比，或者把它作为论文叙事中“NeuroHorizon 严格优于现有模型”的证据，那么 **现阶段证据不够，且表述明显过界**。

我建议把 1.8.3 在当前版本中的定位，正式改写为：

> **探索性 custom baseline study（非正式 benchmark）**

在真正补齐真实模型适配、统一 test eval、统一采样协议之前，不要继续把它用于支撑强结论。

## 2026-03-17 审查回应与修正计划（基于代码验证 + 原始模型调研）

> 本节基于对审计报告所有引用代码的逐行验证，以及对三个原始模型仓库（NDT2 `context_general_bci`、Neuroformer、IBL-MtM）的深度代码调研，给出评估结论和修正行动计划。
> 评估标准：以**忠实的原始模型适配**和**严格的 held-out test 评估**作为唯一合格基准。

### 一、逐条评估

#### Finding 1：简化重写而非真实适配 — 确认成立，升级为阻断性问题

**代码验证结论**：三个 adapter 均为从零构建的简化 Transformer，与原始模型架构存在本质差异：

| 模型 | 当前 adapter 实现 | 原始模型核心特征 | 差异严重度 |
|------|------------------|-----------------|-----------|
| **NDT2** | 单个 `nn.TransformerEncoder` + linear head；无 tokenization；无 context embedding；bidirectional encoder 但仅通过 zero-future 实现 | `SpaceTimeTransformer`：space-time token 分组（每 8 neurons 一个 token）；多层 context 嵌入系统（session token + subject embed + task embed + array embed + space position embed）；MAE/ShuffleInfill 预训练任务；encoder-decoder 架构 | **致命** |
| **Neuroformer** | Binned counts 输入 -> causal Transformer -> predict log-rate per bin；本质是一个标准 GPT 做 binned count regression | 离散 spike token（neuron ID + dt token）-> 逐 spike 自回归生成；同时预测 neuron identity 和 firing time；cross-attention 多模态融合（neural state × history × stimulus × behavior）；precision/recall/F1 评估范式 | **致命** |
| **IBL-MtM** | `nn.TransformerEncoder` + causal mask + linear head；无 session embedding；无 stitching；固定 channel 槽位 | NDT1 架构：`NeuralStitcher`（跨 session neuron 数量对齐）；per-session `nn.Embedding`；多种 masking 策略（temporal / neuron / inter-region / intra-region）+ 对应 prompt token；RoPE attention | **致命** |

**关于"是否可以不做真实适配"的判断**：

不能。原因如下：
1. Neuroformer 的核心创新是**逐 spike 自回归生成**（同时预测 neuron ID 和精确 firing time），而非 binned count regression。当前 adapter 使用 binned counts 输入做 log-rate 预测，本质上是一个标准 causal Transformer，与 Neuroformer 的核心机制完全不同。这不是"简化"，而是**不同的方法**。
2. NDT2 的核心是 ShuffleInfill（MAE 风格 encoder-decoder），而非 "zero future + bidirectional encoder"。后者丢失了 NDT2 论文的核心贡献（见 NDT2 论文 Fig 2 的架构示意）。
3. IBL-MtM 的 multi-task masking（同时训练 temporal / neuron / inter-region / intra-region 四种 mask 任务 + prompt token 路由）是其区别于简单 masked prediction 的核心创新。当前 adapter 仅用了 causal mask 一种模式。

**结论**：当前 1.8.3 的三个 adapter 不能代表 NDT2 / Neuroformer / IBL-MtM，即便加 "style" 后缀也不合适（因为核心机制都不一样），最多只能称为 "3 个项目内 Transformer baseline（bidirectional / causal-GPT / causal-with-auxiliary-loss）"。

#### Finding 2：只在 validation 上评估，没有 test — 确认成立

代码铁证：
- `benchmark_train.py` L48-63 创建了 `test_dataset`、L116 打印了 `Test samples` 数量，但整个训练循环（L141-218）只使用 `val_loader`
- `results.json` 只存储 `best_val_fp_bps`，无任何 test 指标
- NeuroHorizon 自己的 `eval_phase1_v2.py` continuous eval 也在 `split="valid"` 上评估（L112），不是 test

**加严判断**：即便统一了 sampler 和 null model，只要最终报告数字来自 validation（也是 model selection 的依据），就不能称为"最终评估结果"。这是一个实验方法论问题：validation 数字天然偏乐观（因为 checkpoint 就是按它选的）。

**需要同时修复两侧**：benchmark 和 NeuroHorizon 都应在 held-out test split 上报告最终结果。

#### Finding 3：采样器不同 — 确认成立，严重度上调

审计报告指出了样本数差异（benchmark 250ms valid: 1588 vs NeuroHorizon: 841），但未深入分析原因。通过代码验证，差异来源明确：

- `BenchmarkDataset`（base_adapter.py L127-137）：50% overlap 滑窗，步长 = `window * 0.5`
- `RandomFixedWindowSampler`（sampler.py L45-137）：无 overlap，步长 = `window_length`，且有随机 offset

**上调严重度理由**：虽然 fp-bps 的期望值不会因采样策略系统性偏移，但 50% overlap 会导致相邻样本高度相关（前半部分完全重叠），这意味着 benchmark 的 1588 个 validation 样本的有效独立样本数远少于 1588，方差估计会偏低。而 NeuroHorizon 的 841 个无重叠样本则更接近独立采样。两者的**置信区间**不可比。

在要求严格评估的标准下，这不是"小问题"——应当统一为同一个 sampler。

#### Finding 4：best fp-bps 与 last R² 混写 — 确认成立

数据验证：

```
ndt2     250ms: best_fp=0.1691  last_fp=-0.0931  last_r2=0.2138  max_r2=0.2412
neuroformer 250ms: best_fp=0.1856  last_fp=0.0612  last_r2=0.2136  max_r2=0.2426
ibl_mtm  250ms: best_fp=0.1749  last_fp=0.0198  last_r2=0.2101  max_r2=0.2411
```

NDT2 的 last_fp=-0.0931（严重过拟合），此时的 last_r2=0.2138 与 best_fp epoch 的 R² 必然不同。`visualize_benchmarks.py` L155-168 取 `r2_hist[-1]` 而文档写成 "Best R² 汇总"——这是一个确认的 bug。

#### Finding 5：Null model 未统一 — 确认成立，严重度维持为低

代码确认两套不同实现：
- `compute_null_rates()`（neurohorizon_metrics.py L181-243）：遍历训练 intervals 的原始 spike events
- `compute_benchmark_null_rates()`（base_adapter.py L220-244）：遍历 50% overlap 的 binned 样本

**C1. NDT2 真实适配（优先级：高，估计工时：3-5 天）**

适配路径：绕过 NDT2 的 context registry 系统，直接构造 batch dict 送入 `BrainBertInterface`。

主要挑战：NDT2 的 `SpikingDataset` + `DataAttrs` 类型系统较复杂，需要仔细构造匹配的 batch 格式。

**C2. IBL-MtM 真实适配（优先级：高，估计工时：5-7 天）**

适配路径：编写自定义 Dataset 类，对接 NDT1 模型的 forward 接口。

关键步骤：
1. 将 torch_brain spike events 转为 binned counts `[bs, T, N]`
2. 编写 Perich-Miller 的 Dataset 类（参考 `_preprocess_h5_data` 路径）
3. 处理跨 session neuron 数量差异：使用 `NeuralStitcher`（per-session 线性投影到统一维度）
4. 配置 session embedding：为每个 Perich-Miller session 注册 session ID
5. 配置 masking 策略：使用 `causal` 模式（`context_forward=0, context_backward=-1`）做 forward prediction
6. 消除 IBL 硬编码依赖：`target_eids.txt` 模块级读取、`RegionLookup` 依赖 `iblatlas`
7. 评估时使用 `forward_pred` 模式（eval_utils.py L285-370）：mask 未来时间步，读取 causal 预测输出

主要挑战：IBL 硬编码依赖较多（session EID lookup、region atlas），消除这些依赖需要修改源码。NeuralStitcher 需要为每个 session 单独注册。

**C3. Neuroformer 真实适配（优先级：中，估计工时：4-6 天）**

适配路径：编写 Perich-Miller 数据加载函数，使用 Neuroformer 原生的 spike token 化 + 自回归生成。

关键步骤：
1. 编写 `load_perich_miller()` 函数：从 torch_brain 读取 spike events，构建 `data['spikes']` 矩阵 `(N_neurons, N_timesteps)`
2. 编写 YAML 配置：设定 `window`（50ms）, `resolution.dt`（1ms 或 5ms）, `block_size.id`（基于窗口内平均 spike 数量），关闭视觉刺激层（`layers.stimulus = 0`）
3. 使用 `NFDataloader` 做 spike tokenization（neuron ID + dt token，含 SOS/EOS/PAD）
4. 使用 Neuroformer 原生 `forward()` 做训练（cross-entropy loss on neuron ID + dt）
5. 使用 `generate_spikes()` 做自回归推理（逐 spike 采样 neuron ID 和 dt）
6. 评估：将生成的 spike events 重新 binning 为 `[T, N]` counts，送入统一 fp-bps 评估

主要挑战：Neuroformer 的逐 spike 生成推理极慢（每个时间窗口可能有数十到数百个 spike event 需要逐个生成）；生成结果需要后处理（去 padding、re-binning）才能与 binned count 模型的输出在同一口径下比较。此外，没有视觉刺激输入时，Neuroformer 的多模态 cross-attention 架构可能无法发挥其设计优势。

**适配优先级建议**：NDT2 > IBL-MtM > Neuroformer

理由：
- NDT2 的 batch dict 接口相对解耦，适配工程量最小
- IBL-MtM 的 causal forward prediction 评估模式与 NeuroHorizon 的评估场景最接近
- Neuroformer 的逐 spike 生成范式与 binned count 评估之间存在根本性的口径差异，真实适配的公平性存疑

### 三、对审计报告本身的补充意见

1. **Finding 5（null model）的严重度应降为 [低]**：50% overlap 不改变 `mean_rate = total_spikes / total_bins` 的期望值（overlap 导致分子分母同比增加），数值影响可忽略。统一实现是好的工程实践，但不影响结论。

2. **审计报告未覆盖的一个额外问题**：三个 benchmark 模型全部出现了**训练后期严重过拟合**（NDT2 250ms: best_fp=0.169, last_fp=-0.093），说明当前的 learning rate schedule 和 regularization 配置可能不合适。即便作为简化 baseline，这个过拟合程度也提示训练策略需要调整（可能需要 early stopping 或更强的 weight decay）。这在忠实适配原始模型时应格外注意——原始模型的训练策略是经过调优的，而简化重写丢失了这些经验。

3. **审计报告的修正建议 B "重做 benchmark pipeline" 中"显式建模 unit / session identity"的建议需要区分对待**：NDT2 原始论文本身不使用 per-unit embedding（而是通过 space token position 隐式区分），所以"要求所有模型都建模 unit identity"并不符合所有原始模型的设计。正确的做法是：忠实复现每个模型自己的 identity 建模方式（NDT2 用 session token + space position；IBL-MtM 用 session embedding + stitcher；Neuroformer 用 neuron ID token）。

## 2026-03-17 审查回应补充（代码修正后的二次整合）

### 本轮新增代码验证

这轮我把 Claude 补充里“数据适配层可能仍有结构性偏差”的怀疑继续往下追，结果确认了两个比 `heldout_decoding` 更高优先级的 fidelity 问题，而且它们都会直接改变 NDT2 的 benchmark 结果：

1. **旧 faithful bridge 把所有 session 都强行扩成全局 `channel_capacity=72` 的 flat token 序列。**
   - 这不符合上游 `serve_tokenized_flat` 的实际行为。
   - Perich-Miller 10 sessions 的真实 token 数应随 session unit 数变化：
     - `71 units -> 9 tokens/step`
     - `52 units -> 7 tokens/step`

     - `19/18 units -> 3 tokens/step`

### 当前负结果的可能原因（假设清单）

下面这些是当前最可能的解释，不应写成已经证实的定论：

1. **训练目标与 benchmark 目标不一致**
   - 当前 faithful NDT2 保留的是上游 `ShuffleInfill`
   - 但 benchmark 评估的是固定 `obs_window -> pred_window` 的 forward prediction
   - 当前失败更像是 objective mismatch，而不是单一实现 bug

2. **train-time 采样分布与 NeuroHorizon 主线不同**
   - 当前 faithful NDT2 已统一数据源 / split / eval / metric
   - 但训练仍不是 NeuroHorizon 的 `RandomFixedWindowSampler`
   - 这不是当前主问题，但仍是一个未控差异项

3. **NDT2 的原始 inductive bias 可能不适合当前 canonical forecasting benchmark**
   - faithful reproduction 解决的是“是不是原模型”
   - 不保证解决“是不是当前 benchmark 的合适模型”

4. **mixed-session flat token batching 可能仍然偏难**
   - Perich-Miller 10 sessions 的 token 数从 `3` 到 `9` 不等
   - 即使 tokenization / padding 已修复，mixed-session batching 本身仍可能让 NDT2 更难收敛

5. **compatibility bridge 仍然不是上游 repo 的原生训练生态**
   - 当前实现已经足够 faithful
   - 但仍不是直接在上游原生 dataset/task pipeline 中训练
   - 因此仍可能存在会累积成性能损失的 subtle mismatch

6. **上游训练规程直迁失败，说明“继续调 optimizer”不是主因**
   - `configure_optimizers() + accumulate=16 + warmup-cosine` 已经尝试过
   - 结果明显更差
   - 现阶段继续围绕 warmup / accumulate 做局部调参，预期收益较低

   - unified Poisson/count metrics 对 token-level generation errors 的惩罚非常重，可能把 sequence mismatch 成倍放大

### 当前 1.8.3 的最新阶段判断（更新）

1. 目前 **三个 faithful bridge（NDT2 / IBL-MtM / Neuroformer）都已经至少打通到 smoke 或 debug train 阶段**
2. 但当前 **没有任何一个 faithful model** 已经在 canonical 250ms benchmark 上给出可接受结果
3. 因此 1.8.3 仍然只能处于：
   - `faithful implementation progressively landing`
   - `benchmark conclusion still invalid / not ready for main table`
4. 后续优先级仍然应该是：
   - IBL-MtM：检查 explicit forward-pred masking 与 multi-task masking fidelity 的差异，再做更长训练
   - Neuroformer：检查 no-vision compatibility、block-size / dt token 设定，以及 generation-to-count eval mismatch
   - 在 IBL-MtM / Neuroformer 至少有一条线从 debug negative stage 走出之前，不把 250ms 结果写回正式 benchmark 主结论，更不扩 500ms / 1000ms

### 2026-03-18 严格适配复核与 250ms 先行验证（更新）

#### 复核结论

1. **NDT2：当前实现满足“统一 protocol + 保留上游核心训练语义”的要求。**
   - 数据源 / split / held-out eval / metric 继续走 `repro_protocol`
   - 训练端继续保留上游 `BrainBertInterface + ShuffleInfill`

2. **IBL-MtM：已从“不合要求”修正为“训练保留 upstream SSL multi-mask，评估单独做 one-step fp”。**
